## ユーザ登録について

`Web` アプリケーションでは、新しい利用者が自分でアカウントを作成できるように、`ユーザ登録機能` を用意することが多いです。  

### 登録機能
この章では登録フォーム、入力値の受け取り、重複チェック、`DB` 登録、登録後のセッション保存までの流れを学びます。

| 項目 | 内容 |
| ---- | ---- |
| 登録フォーム | アカウント名、表示名、メールアドレス、パスワードを入力する画面 |
| 重複チェック | 既に使われているアカウント名やメールアドレスを確認する処理 |
| セッション | 入力値やエラーメッセージ、ログイン状態を一時的に保持する仕組み |
| ユーザ登録 | 入力値を `users` テーブルに保存する処理 |


### アプリ設計
今回の登録画面では単にフォームを作るだけでなく、`Web` アプリの基本的な考え方がいくつも入っています。

| 考え方 | 内容 |
| ---- | ---- |
| 共通化 | `app.php` に共通設定をまとめる |
| 分担 | 画面表示、登録処理、モデル処理を分ける |
| 安全性 | サニタイズ、ハッシュ化、プリペアドステートメントを使う |
| 利便性 | 入力保持やエラー表示で使いやすくする |
| 状態管理 | セッションで入力値やログイン情報を保持する |

### 登録処理の全体像

登録機能は、次の順番で動きます。

| 手順 | 内容 |
| ---- | ---- |
| 1 | `regist/input/` でフォームを表示する |
| 2 | ユーザが必要項目を入力して送信する |
| 3 | `regist/add/` で入力値を受け取る |
| 4 | 入力値をサニタイズしてセッションに一時保存する |
| 5 | 既存ユーザがいないか確認する |
| 6 | 問題なければ `DB` に登録する |
| 7 | 登録したユーザ情報をセッションに保存する |
| 8 | `regist/result/` にリダイレクトして完了画面を表示する |

## アプリ共通処理
会員登録だけに限らず、Webアプリではアクセス時に必ず同じ処理をすることが多くあります。
今回の `app.php` は、**どのページでも最初に読み込む共通ファイル**で、セッション開始、定数定義、ライブラリとモデルの読み込みをまとめています。

```php
// 環境設定読み込み
require_once "env.php";

// セッション開始
session_start();
session_regenerate_id(true);

// アプリベースディレクトリパス
const BASE_DIR = __DIR__;
const APP_DIR = __DIR__ . "/app/";
const LIB_DIR = __DIR__ . "/lib/";
const COMPONENT_DIR = __DIR__ . "/app/components/";

// ライブラリ、モデルなどの事前読込み
require_once LIB_DIR . 'Database.php';
require_once LIB_DIR . 'Sanitize.php';
require_once LIB_DIR . 'File.php';
require_once LIB_DIR . 'Model.php';

require_once APP_DIR . 'models/User.php';
require_once APP_DIR . 'models/AuthUser.php';
```

### app.php にまとめる理由

| 理由 | 内容 |
| ---- | ---- |
| 再利用しやすい | 各ページが同じ設定を使える |
| 書き漏れを防げる | セッション開始やクラス読み込みを毎回書かなくてよい |
| 見通しがよい | 各ページは本来の処理だけに集中できる |

---

### セッションの使い方

登録機能では、`セッション` を主に3つの目的で使っています。

| 用途 | 保存している内容 |
| ---- | ---- |
| 入力保持 | `$_SESSION[APP_KEY]['regist']` に直前の入力値を保存する |
| エラー表示 | `$_SESSION[APP_KEY]['errors']` にエラーメッセージを保存する |
| ログイン状態 | `$_SESSION[APP_KEY]['auth_user']` に登録後のユーザ情報を保存する |

### APP_KEY の役割

`APP_KEY` は、セッションの保存先キーとして使う定数です。

```php
const APP_KEY = "mysecret";
```

これを使うことで、アプリごとのデータを `$_SESSION` の中でまとめて管理できます。

```php
$_SESSION[APP_KEY]['regist'] = $posts;
$_SESSION[APP_KEY]['errors']['public'] = 'このアカウント名は既に使用されています。';
$_SESSION[APP_KEY]['auth_user'] = $auth_user;
```

> 同じサーバで複数の `PHP` アプリを動かす場合でも、キーを分けておくと衝突しにくくなります。

## 登録フォームの表示

登録フォーム画面、フォームを表示する前に前回の入力値とエラーメッセージを取り出します。

#### regist/input/index.php
```php
// 前回の入力値
$regist = [];
if (isset($_SESSION[APP_KEY]['regist'])) {
    $regist = $_SESSION[APP_KEY]['regist'];
}
// エラーメッセージ
$errors = [];
if (isset($_SESSION[APP_KEY]['errors'])) {
    $errors = $_SESSION[APP_KEY]['errors'];
    unset($_SESSION[APP_KEY]['errors']);
}
```

### フラッシュメッセージ

エラーメッセージは表示したらすぐ `unset()` しています。これは `フラッシュメッセージ` の考え方で、1回だけエラー表示する手法です。

| 処理 | 内容 |
| ---- | ---- |
| 保存 | エラー発生時にセッションへ入れる |
| 表示 | 入力画面で取り出して表示する |
| 削除 | 次の画面表示では残らないように消す |

### 入力値の再表示

入力画面では、直前に送信した値を `value` 属性に入れています。
これによりエラーで入力画面へ戻ったときも、**前回の入力がキープ**されます。

```php
<input type="text" name="account_name" value="<?= $regist['account_name'] ?? '' ?>">
```

> パスワードは再表示しないのが一般的です。

## サニタイズ
### サニタイズとは
Webアプリには不特定多数のユーザーが文字を入力しますが、中には悪意を持って**プログラムとして意味を持つ特殊な文字**を混ぜる人がいます。これをそのまま処理してしまうと、アプリが乗っ取られたり、データが盗まれたりする攻撃（SQLインジェクションやXSSなど）を許してしまいます。

### サニタイズの目的

| 目的 | 内容 |
| ---- | ---- |
| 前後の空白を除去する | `trim()` で不要な空白を削除する |
| `HTML` 特殊文字を無害化する | `htmlspecialchars()` でタグ解釈を防ぐ |
| 入力データを整える | 後続の登録処理で扱いやすくする |

> サニタイズは大切ですが、`DB` への安全な保存には引き続き `PDO` のプリペアドステートメントを使います。

`regist/add/index.php` では、最初に `$_POST` を `sanitize()` しています。

```php
$posts = sanitize($_POST);
```

`lib/Sanitize.php` の `sanitize()` は配列を再帰的に処理し、`trim()` と `htmlspecialchars()` を使って値を整えます。

#### lib/Sanitize.php
```php
function sanitize($data)
{
    if (is_array($data)) {
        return array_map('sanitize', $data);
    }
    return trim(h($data, ENT_QUOTES, 'UTF-8'));
}
```

## 登録処理

`regist/add/index.php` が、実際の登録処理を担当しています。

```php
if ($_SERVER['REQUEST_METHOD'] !== 'POST') {
    exit;
}

$posts = sanitize($_POST);
$_SESSION[APP_KEY]['regist'] = $posts;
```

### POST のみ受け付ける理由

登録処理はデータを書き込む処理なので、`GET` アクセスで実行されないように `POST` のみ受け付けます。

| メソッド | 用途 |
| ---- | ---- |
| GET | 画面表示、データ取得 |
| POST | 登録、更新、削除などデータ変更 |

---

### 重複チェック

登録前に同じアカウント名やメールアドレスが既に使われていないか確認します。

```php
$user = new User();
$user_exists = $user->findForExists($posts);

if (!empty($user_exists['id'])) {
    $_SESSION[APP_KEY]['errors']['public'] = 'このアカウント名は既に使用されています。';
    header('Location: ../input/');
    exit;
}
```

### findForExists() の処理

`User` モデルでは、`account_name` または `email` が一致するユーザを検索しています。

```php
$sql = "SELECT * FROM users WHERE account_name = :account_name OR email = :email";
$stmt = $pdo->prepare($sql);
$stmt->execute([
    'account_name' => $account_name,
    'email' => $email
]);
```

### なぜ事前チェックするのか

| 理由 | 内容 |
| ---- | ---- |
| 重複登録を防ぐ | 同じアカウント名やメールアドレスを登録させない |
| わかりやすいエラーにできる | 利用者に理由を伝えやすい |
| `DB` 制約違反を減らせる | `UNIQUE` エラーを未然に防ぎやすい |

---

### ユーザデータ登録処理

ユーザデータの `DB` 登録は `User` モデルの `insert()` が担当します。

#### app/models/User.php
```php
public function insert($data)
{
    // パスワードハッシュ化
    $data['password'] = password_hash($data['password'], PASSWORD_DEFAULT);

    $pdo = Database::getInstance();
    // ユーザ登録のSQL
    $sql = "INSERT INTO users (account_name, email, display_name, password)
            VALUES (:account_name, :email, :display_name, :password)";

    $stmt = $pdo->prepare($sql);
    $result = $stmt->execute($data);
    if ($result) {
        return $pdo->lastInsertId();
    }
}
```

### 登録処理のポイント

| ポイント | 内容 |
| ---- | ---- |
| パスワードのハッシュ化 | `password_hash()` で平文のまま保存しない |
| プリペアドステートメント | `prepare()` と `execute()` で安全に登録する |
| 登録直後のID取得 | `lastInsertId()` で新しいユーザIDを得る |

### password_hash() とは

`password_hash()` は、パスワードを元に戻せない文字列である**ハッシュ値**へ変換する関数です。

```php
$hash = password_hash('secret123', PASSWORD_DEFAULT);
```

`DB` に保存されるのは元のパスワードではなく、このハッシュ値です。

> パスワードは絶対にそのまま保存してはいけません。


### 登録後にユーザ情報取得

登録に成功した後は、もう一度ユーザを検索しています。

```php
$user_id = $user->insert($posts);
$auth_user = $user->find($user_id);
```

これは、登録直後のユーザ情報を取り出して、そのままログイン状態として使うためです。

| 取得する理由 | 内容 |
| ---- | ---- |
| セッション保存に使う | 登録完了後にログイン済み状態を作れる |
| 最新の `DB` データを使える | 実際に保存された内容を基準にできる |

## セッション処理

登録成功時はユーザ情報をセッションに保存し、登録完了後のどのページでも**ログイン状態をキープ**します。

#### regist/add/index.php
```php
// セッション保存
$_SESSION[APP_KEY]['auth_user'] = $auth_user;
// リダイレクト
header('Location: ../result/');
exit;
```

### リダイレクト
登録処理後は入力画面または登録完了画面などにリダイレクトします。

| 理由 | 内容 |
| ---- | ---- |
| 二重送信を防ぎやすい | 再読み込みで再登録されにくくなる |
| 画面の役割を分けられる | 処理ページと表示ページを分離できる |
| `URL` がわかりやすい | 入力画面、完了画面を明確に分けられる |

---

## 登録完了画面

登録完了画面では、不要になったセッションデータを整理しています。

```php
if (isset($_SESSION[APP_KEY]['regist'])) {
    unset($_SESSION[APP_KEY]['regist']);
}

if (isset($_SESSION[APP_KEY]['errors'])) {
    unset($_SESSION[APP_KEY]['errors']);
}
```

### なぜ削除するのか

| 削除するデータ | 理由 |
| ---- | ---- |
| regist | 完了後に古い入力値を残さないため |
| errors | 完了画面に不要なエラーを残さないため |

## まとめ

| 項目 | 内容 |
| ---- | ---- |
| 登録フォーム | `regist/input/index.php` で表示し、入力値を再表示できる |
| 登録処理 | `regist/add/index.php` で `POST` を受け取り、登録を行う |
| 重複チェック | `findForExists()` で既存ユーザを確認する |
| パスワード保存 | `password_hash()` を使って安全に保存する |
| セッション | 入力値、エラー、認証ユーザを保持する |
| 完了画面 | `regist/result/index.php` で登録完了を表示し、不要なセッションを削除する |
