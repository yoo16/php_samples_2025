## 5：ルーティング設計とクリーンURL

### 講義（20分）

**なぜクリーンURLが必要か**

| 観点 | 従来URL | クリーンURL |
|---|---|---|
| 見た目 | `user/?id=5` | `user/5` |
| SEO | パラメータは評価されにくい | パスは評価される |
| UX | 意味が分かりにくい | 直感的 |
| フレームワーク | PHPの慣習 | Laravelなどの標準 |

**現在の仕組み（ディレクトリベース）**

```
GET /user/?id=5
  → user/index.php が実行される
  → $_GET['id'] = 5 を取得
```

**クリーンURLの仕組み（フロントコントローラー）**

```
GET /user/5
  → 実在するディレクトリ・ファイルがない
  → .htaccess が router.php に転送
  → router.php がパスを解析して $_GET['id'] = 5 をセット
  → UserController::index() を呼ぶ
```

**なぜ既存コードを変えなくて済むのか**

`router.php` が `$_GET['id'] = 5` をセットするため、
コントローラーは従来通り `$_GET['id']` を読むだけでよい。

## 6：機能を自分で設計する

### 講義（20分）

**良い仕様書の条件**

1. **何を作るか**が明確（概要）
2. **DBの変更**が明示されている（テーブル・カラム）
3. **追加・変更ファイル**の一覧がある
4. **APIの入出力**が具体的（リクエスト・レスポンス例）
5. **実装順序**が書かれている

**設計の考え方：機能をDBから考える**

```
機能アイデア：「ブックマーク機能」
  ↓
必要なテーブル: bookmarks (id, user_id, tweet_id)
  ↓
必要なAPI: api/bookmark/toggle/
  ↓
UIの変更: tweet_nav にブックマークボタン追加
  ↓
仕様書に落とす
```

**既存機能（Like）を参考にする**

新機能を設計するときは、似た既存機能のコードを参照する。
Like機能とBookmark機能は構造がほぼ同じ → Like をベースに設計できる。

### 演習（30分）

**自分で機能を1つ設計して仕様書を書く**

以下の候補から1つ選ぶ（または自分で考える）：

| 機能 | 難易度 |
|---|---|
| ブックマーク | ★☆☆（Likeとほぼ同じ構造） |
| ツイート編集 | ★★☆（既存のinsertをupdateに変える） |
| 通知機能 | ★★★（複数のアクションをまとめて管理） |
| ハッシュタグ一覧 | ★★☆（検索機能の応用） |

**仕様書のテンプレート**

```markdown
# [機能名] 仕様書

## 概要
（何ができるようになるか、1〜2行で）

## DBの変更
（新規テーブルまたはカラム追加のSQL）

## 追加・変更ファイル一覧
（新規作成 / 変更 / 変更しない）

## API設計
（エンドポイント・メソッド・リクエスト・レスポンス）

## 実装順序
1. 〜
2. 〜
```
